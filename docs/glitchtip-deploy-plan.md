## Two-phase rollout

**Phase 1: tailnet-only smoke.** Helm-install with `ingress.enabled: false` and `service.annotations` carrying `tailscale.com/expose: "true"` + `tailscale.com/hostname: glitchtip`. Smoke-test over tailnet only: log in, create a project, point a throwaway Sentry SDK at the DSN, fire a synthetic error, see it land in the UI.

**Phase 2: public.** Once tailnet smoke passes (login + DSN ingest + UI render), add the Route 53 A record, flip `ingress.enabled: true` in the values file, swap `GLITCHTIP_DOMAIN` to `https://glitchtip.coilysiren.me`, and `helm upgrade`. Tailnet Service stays exposed as a backstop.

This keeps Route 53 + cert-manager DNS-01 + public ingress off the table until GlitchTip is proven to work end-to-end inside the trust boundary.

**Phase 3 (separate, not blocking either phase above).** Swap Sentry DSNs in `backend`, `eco-mcp-app`, `eco-spec-tracker`, `eco-telemetry`, `coily` to the GlitchTip ones, one app at a time. Mailer wiring (SMTP via SES or similar) lands here too if password reset over email turns out to be needed.

## SSM secrets to mint (1 on phase 1, +1 after admin rotation)

One `coily aws ssm put-parameter --type SecureString` call under `/glitchtip/*`. Generate with `openssl rand` and pipe via `--value file:///dev/stdin` so the secret never lands in argv. Update `coilyco-ai/SSM.md` in the same logical change set with a `/glitchtip/*` bullet.

| SSM path | Length / format | GlitchTip env binding |
|---|---|---|
| `/glitchtip/secret-key` | 64 char hex (`openssl rand -hex 32`) | `SECRET_KEY` (Django) |
| `/glitchtip/admin-password` | 32 char alnum, generated in-pod first, then rotated in UI | Not bound to env. Holds the rotated value for password-manager retrieval. Written after step 7 of the phase-1 apply order. |

Canonical write shape (no value in argv):

```sh
openssl rand -hex 32 \
  | coily --commit-scope=infrastructure aws ssm put-parameter \
      --name /glitchtip/secret-key \
      --type SecureString \
      --value file:///dev/stdin
```

`/glitchtip/admin-password` is not a Postgres or service password. Postgres credentials are auto-generated by the bundled `postgresql` subchart and stored in the chart-managed Secret. No SSM round-trip for those.

## Manifest shape

Three files. Helm-installed, so the bulk of resources are chart-rendered, not hand-written.

**1. `infrastructure/deploy/observability/glitchtip-externalsecret.yml`**

Single ExternalSecret pulling `/glitchtip/secret-key` into a k8s Secret named `glitchtip-django-secret`, in the `observability` namespace, against the existing `aws-parameter-store` ClusterSecretStore. Mirrors `forgejo-secrets`' shape.

**2. `infrastructure/deploy/observability/glitchtip-values.yml`** (the helm values file)

Doc-order keys, all version-pinned:

- `image.tag`: pinned to the chart's bundled app version. Don't float.
- `env`:
  - `SECRET_KEY` via `existingSecret: glitchtip-django-secret`, key `secret-key`
  - `GLITCHTIP_DOMAIN`: phase 1 `http://glitchtip.<tailnet>.ts.net`, phase 2 `https://glitchtip.coilysiren.me`
  - `DEFAULT_FROM_EMAIL`: `glitchtip@coilysiren.me` (placeholder, mailer disabled)
  - `EMAIL_URL`: unset
  - `ENABLE_OPEN_USER_REGISTRATION`: `"false"`
  - `CELERY_WORKER_AUTOSCALE`: `"1,3"` (low ceiling for a homelab)
- `web`:
  - `replicaCount: 1`
  - `resources.requests`: `cpu: 200m`, `memory: 512Mi`
  - `resources.limits`: `cpu: 1000m`, `memory: 1500Mi`
  - `service.annotations`:
    - `tailscale.com/expose: "true"`
    - `tailscale.com/hostname: glitchtip`
- `worker`:
  - `replicaCount: 1`
  - `resources.requests`: `cpu: 100m`, `memory: 256Mi`
  - `resources.limits`: `cpu: 500m`, `memory: 768Mi`
- `migrationJob.enabled: true` (chart default, runs `manage.py migrate` as a Job on every release)
- `ingress.enabled: false` (phase 1, flipped in phase 2)
- `postgresql`:
  - `enabled: true`
  - `auth.database: glitchtip`
  - `auth.username: glitchtip`
  - `primary.persistence.enabled: true`
  - `primary.persistence.size: 10Gi`
  - `primary.persistence.storageClass: local-path`
  - `primary.resources.requests`: `cpu: 100m`, `memory: 256Mi`
  - `primary.resources.limits`: `cpu: 500m`, `memory: 768Mi`
- `valkey`:
  - `enabled: true`
  - `master.persistence.enabled: false`
  - `replica.replicaCount: 0` (no HA, single master is fine)
  - `auth.enabled: true` (chart default, password auto-generated into chart-managed Secret)

The exact key paths in the chart's `values.yaml` need a sanity check against chart 8.2.0 before commit (chart key names drift across major versions). Reference: `helm show values glitchtip/glitchtip --version 8.2.0`.

**3. `infrastructure/deploy/observability/README.md`** (existing file, append a section)

## Apply path

**Pattern: commit local → push to GitHub → SSH to kai-server → `git pull` → `helm install` / `helm upgrade`.** Same shape as the observability stack README's "First-time install" section, and the same SSH-based apply discipline the forgejo plan settled on.

`coily ssh kubectl` blocker from the forgejo plan turned out to be solvable per the [coily#56](https://github.com/coilysiren/coily/issues/56) retro note: `k3s kubectl ...` directly via `ssh kai-server '...'` works without sudo because `/etc/rancher/k3s/k3s.yaml` is mode 644. Use that for watching from this Mac.

## Apply order - phase 1 (tailnet-only)

1. Add the helm repo locally:

    ```sh
    helm repo add glitchtip https://gitlab.com/api/v4/projects/16325141/packages/helm/stable
    helm repo update
    helm show values glitchtip/glitchtip --version 8.2.0 > /tmp/glitchtip-defaults.yml
    ```

    Sanity-check the key paths used in `glitchtip-values.yml` against `/tmp/glitchtip-defaults.yml`. Chart key names drift across majors. Fix divergences before commit.

2. Mint the SSM param via the stdin pipe pattern. Verify with `coily aws ssm describe-parameters --filters Key=Name,Values=/glitchtip/`.

3. Update `coilyco-ai/SSM.md` with a `/glitchtip/*` bullet (one key for now, `secret-key`). Commit + push (coilyco-ai repo).

4. Write `infrastructure/deploy/observability/glitchtip-externalsecret.yml` and `glitchtip-values.yml`. Update `deploy/observability/README.md` with the install + upgrade commands. Commit + push (infrastructure repo).

5. SSH to kai-server. `cd` into the infrastructure clone there, `git pull`, then:

    ```sh
    k3s kubectl apply -f deploy/observability/glitchtip-externalsecret.yml
    # Wait for ExternalSecret to sync the k8s Secret before helm-install,
    # otherwise the web pod's env binding fails at startup.
    until k3s kubectl -n observability get secret glitchtip-django-secret >/dev/null 2>&1; do sleep 2; done
    helm install glitchtip glitchtip/glitchtip \
      --version 8.2.0 \
      --namespace observability \
      -f deploy/observability/glitchtip-values.yml
    ```

6. Watch the rollout from the same SSH session:

    ```sh
    k3s kubectl -n observability get pods,jobs,externalsecret -l app.kubernetes.io/instance=glitchtip -w
    ```

    Wait for `glitchtip-postgresql-0` Ready, then `glitchtip-valkey-master-0` Ready, then the `glitchtip-migrate` Job Complete, then `glitchtip-web` and `glitchtip-worker` Ready.

7. Create the admin user via non-interactive `createsuperuser`. Generate `$INITIAL` outside the pod and pass via Django's `DJANGO_SUPERUSER_*` env vars so it never lands in argv:

    ```sh
    INITIAL=$(openssl rand -base64 24 | tr -d '=+/' | head -c 32)
    k3s kubectl -n observability exec deploy/glitchtip-web -- \
      env DJANGO_SUPERUSER_USERNAME=coilysiren \
          DJANGO_SUPERUSER_EMAIL=coilysiren@gmail.com \
          DJANGO_SUPERUSER_PASSWORD="$INITIAL" \
      python manage.py createsuperuser --no-input
    ```

    Capture `$INITIAL` for step 8 only. Don't echo it to chat.

8. **Browser automation via playwright-mcp.** Same shape as the forgejo phase-1 step 8:
    - Navigate to `http://glitchtip.<tailnet>.ts.net/login`, log in as `coilysiren@gmail.com` / `$INITIAL`.
    - Settings → Account → Change Password. Generate `$ROTATED = openssl rand -base64 24`, fill old / new, submit.
    - Organizations → New Organization, name `coilysiren`, submit.
    - Projects → New Project, name `smoke`, platform Python, submit. Capture the DSN that the UI hands back.
    - Print `$ROTATED` and the smoke-project DSN to chat once at the end. Kai files the password in her password manager and the DSN goes into step 9.
    - Then mint `/glitchtip/admin-password` via the same stdin-pipe SSM pattern, with `$ROTATED` as the value.

    Ref-handle caveat carried over from the forgejo plan: after every navigation, re-call `browser_snapshot` before the next `fill_form` / `click`. Don't reuse refs across navigations.

9. **End-to-end smoke** (must pass before phase 2). Driven from this Mac:

    ```sh
    uv run --with sentry-sdk python -c "
    import sentry_sdk, os
    sentry_sdk.init(dsn=os.environ['SMOKE_DSN'])
    1/0
    " 2>/dev/null || true
    ```

    With `SMOKE_DSN` set to the project DSN from step 8. Confirm the `ZeroDivisionError` appears in the GlitchTip UI within ~10s. Delete the `smoke` project from the UI afterward.

10. Schedule the post-push CI check per `infrastructure/AGENTS.md` (300s after push, re-check at +180s if in progress). Infra CI is pylint-only but the rule still applies.

## Apply order - phase 2 (public)

Only after phase-1 smoke passes.

1. **Route 53 A record.** Add `glitchtip.coilysiren.me → 99.110.50.213` in zone `Z06714552N3MO04UBWF33`:

    ```sh
    coily --commit-scope=infrastructure aws route53 change-resource-record-sets \
      --hosted-zone-id Z06714552N3MO04UBWF33 \
      --change-batch '{
        "Changes": [{
          "Action": "CREATE",
          "ResourceRecordSet": {
            "Name": "glitchtip.coilysiren.me.",
            "Type": "A",
            "TTL": 300,
            "ResourceRecords": [{"Value": "99.110.50.213"}]
          }
        }]
      }'
    ```

    Verify with `dig +short glitchtip.coilysiren.me`. Should return `99.110.50.213` within ~5 minutes.

2. **Edit `glitchtip-values.yml`**:
    - Flip `ingress.enabled: true`.
    - Add `ingress.hostname: glitchtip.coilysiren.me`, `ingress.ingressClassName: traefik`, `ingress.tls: [{hosts: [glitchtip.coilysiren.me], secretName: glitchtip-tls}]`, annotations `cert-manager.io/cluster-issuer: letsencrypt-production`, `kubernetes.io/tls-acme: "true"`.
    - Swap `env.GLITCHTIP_DOMAIN` to `https://glitchtip.coilysiren.me`.

3. **Commit + push.** Suggested subject:

    > glitchtip: phase-2 public Ingress (cert-manager DNS-01)

4. **Pull + upgrade on kai-server:**

    ```sh
    ssh kai-server 'cd /home/kai/projects/coilysiren/infrastructure && git pull --ff-only && \
      helm upgrade glitchtip glitchtip/glitchtip \
        --version 8.2.0 \
        --namespace observability \
        -f deploy/observability/glitchtip-values.yml'
    ```

5. **Watch the certificate** (DNS-01 takes 1-2 min on first issuance):

    ```sh
    ssh kai-server 'k3s kubectl -n observability get certificate glitchtip-tls -w'
    ```

    Wait for `READY=True`. If it stalls, `describe order` / `describe challenge` per `k3s-deploy-notes.md`.

6. **Re-run the smoke against the public hostname.** Same `1/0` Python one-liner from phase-1 step 9, with `SMOKE_DSN` swapped to the project DSN's public host. Confirm UI receives the event.

7. **Verify TLS chain:**

    ```sh
    curl -sI https://glitchtip.coilysiren.me/ | head -3
    openssl s_client -connect glitchtip.coilysiren.me:443 -servername glitchtip.coilysiren.me </dev/null 2>/dev/null \
      | openssl x509 -noout -issuer -subject -dates
    ```

    Issuer Let's Encrypt R10/R11. `notAfter` ~90 days out.

8. **Schedule the post-push CI verify** per `infrastructure/AGENTS.md`.

## Phase 2 rollback

If the cert never issues, or the public surface breaks unrecoverably:

1. Edit `glitchtip-values.yml` back to `ingress.enabled: false`, `GLITCHTIP_DOMAIN` back to the tailnet URL. `helm upgrade` reverts.
2. Revert the commit, push, re-pull on the server.
3. The tailnet front door (`http://glitchtip.<tailnet>.ts.net/`) keeps working throughout. Nothing in phase 2 touches the tailscale-annotated Service.

The Route 53 record is safe to leave in place. Without the Ingress it just resolves to a public IP that returns nothing on port 443.

- **Mailer**. Wire SMTP (SES creds via SSM) when password reset over email becomes worth having.
- **Backup CronJob**. Weekly `pg_dump` against the bundled Postgres, writing to a sibling PVC. Drill restore before treating durable.
- **vmalert wiring**. If GlitchTip emits Prometheus metrics, scrape them via the existing `vmagent` and add an alert rule for event-ingest-rate dropping to zero.

## Traps to remember (from k3s-deploy-notes.md and the forgejo retro)

- `hostAliases` for `*.coilysiren.me` → `192.168.0.194` if GlitchTip ever needs to call back into the cluster (webhooks to the eco-telemetry receiver, for example). Add to the chart's `extraPodSpec` if and when needed.
- DNS-01 cert takes 1-2 min on first issuance.
- StatefulSet `serviceName` must match the headless Service name. The bundled `postgresql` subchart gets this right. If it ever doesn't, pods won't get DNS.
- Bundled Postgres auto-generates its password into a chart-managed Secret. Don't reach for that Secret value. Chart owns it.
- Helm chart majors drift key names. Pin both `--version` on `helm install` and revalidate values against `helm show values` on every chart bump.
- `K8S_SERVER` for any CI must be the tailnet IP literal, not `kai-server` (MagicDNS doesn't resolve from GitHub runners). See `k3s-deploy-notes.md §"K8S_SERVER"`.
- ExternalSecret sync timing: `helm install` will start pods before the Secret exists if you don't wait. The phase-1 step-5 `until` loop covers this.
